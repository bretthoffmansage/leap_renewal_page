# REUSABLE_SHELL_GENERATED_DRIFT_RULES

## Purpose

This document defines how the reusable autonomous shell must detect, classify, and handle recurring generated-file drift.

Its job is to prevent the shell from:

- carrying noisy generated changes between packages
- committing unintended generated files as package work
- getting stuck on recurring dirty-tree issues
- forcing the user to manually clean known drift over and over

This is a shell runtime hygiene contract.

It is not product behavior.

---

## Core rule

If a file changes repeatedly as a side effect of build/dev/tooling behavior, and that change is not intentional package work, the shell must treat it as **generated drift**.

Generated drift must be handled automatically where safe.

The shell must not:
- ignore it blindly
- commit it casually
- ask the user to manually clean it every time
- let it keep the tree dirty between completed packages

---

## Why this matters

During GoTeamGo execution, recurring generated drift created avoidable friction.

The most visible example was:

- `next-env.d.ts`

This file repeatedly changed during package execution and validation even when the package did not intentionally modify it.

The reusable shell must treat this kind of behavior as a known runtime concern, not a special manual-case exception.

---

## Definition of generated drift

A file change qualifies as generated drift when all or most of the following are true:

1. the file is generated by framework/tooling behavior
2. the file changes repeatedly across otherwise unrelated units of work
3. the package did not intentionally modify the file as part of its scope
4. the file’s changed contents do not represent a real product/code decision for the package
5. leaving the file dirty would reduce package closeout cleanliness or resume reliability

If these conditions hold, the shell should prefer reverting the file before package closeout.

---

## Definition of intentional generated-file change

Not every generated file change is drift.

A generated file change may be intentional if:

- the package explicitly requires changing generation behavior
- the generated file is part of a real tooling/config migration
- the generated file is intentionally tracked and updated as part of the completed unit
- the package scope clearly includes the file

The shell must classify the file change before acting.

---

# Generated drift handling goals

The shell’s generated-drift behavior should ensure:

1. completed packages commit only intended work
2. known generated noise does not accumulate
3. dirty-tree false positives are reduced
4. resume behavior remains reliable
5. users do not have to manually revert the same files repeatedly

---

# Known recurring drift examples

## 1. `next-env.d.ts`

### Status
Known recurring generated drift candidate.

### Why it matters
- often changes during Next.js build/dev flows
- commonly not part of package intent
- can keep the tree dirty after an otherwise completed package

### Default shell treatment
- inspect whether the package intentionally required this change
- if not intentional, revert it before commit
- re-check `git status` afterward

## 2. Future generated drift candidates

The shell should be designed to handle additional known-drift files later, such as:

- generated route/type files
- framework-generated env/type helpers
- generated artifacts tied to local tooling state
- package manager noise if clearly unrelated to the package
- generated build metadata files

These should be added to the shell’s known-drift list only after they are observed and classified truthfully.

# Classification rules

When a dirty tree exists, the shell must classify each suspicious changed file.

## Classification categories

### Category A — Intentional package work
The file belongs to the package scope and should be committed.

### Category B — Known generated drift
The file changed because of tooling/framework behavior and should be reverted unless intentionally in scope.

### Category C — Ambiguous
It is unclear whether the file belongs to the package or is generated drift.

### Category D — Unrelated leftover work
The file belongs to some other unresolved or accidental state.

The shell must not treat all dirty files the same.

---

# Required generated-drift handling sequence

When closing a completed package or hardening pass, the shell must perform this sequence.

## Step 1 — Inspect dirty state
Run at minimum:
- `git status`
- `git diff --stat`

Purpose:
- identify changed files
- identify untracked files
- identify drift candidates

---

## Step 2 — Check known generated-drift list
The shell should compare dirty files against its known-drift rules.

If a file matches a known-drift candidate, the shell must still ask:

- was this file intentionally changed by this package/pass?

If no, treat it as drift.

---

## Step 3 — Classify file as intentional or drift
For each known drift candidate:
- determine if the package/pass explicitly required this file change
- if yes, keep it
- if no, revert it

The shell must not revert intentional package work just because the file is commonly noisy.

---

## Step 4 — Revert non-intentional drift
If a file is classified as non-intentional generated drift, the shell must revert it before staging/commit.

After reverting, the shell must re-check git state.

---

## Step 5 — Verify clean residual state
After reverting drift, the shell must confirm that remaining dirty files actually belong to the completed unit of work.

Only then may it proceed with staging and commit.

---

# Required behavior for `next-env.d.ts`

The shell must include an explicit rule for `next-env.d.ts`.

## Default rule
If `next-env.d.ts` changes during package execution or hardening and the package/pass did not intentionally modify it:

- revert it automatically before closeout
- do not commit it
- confirm the tree is cleaner afterward

## Exception
If a package specifically and intentionally includes `next-env.d.ts` as part of a real framework/config shift, the shell may keep it — but only if that intent is explicit and justified by the unit of work.

---

# Dirty-tree resolution rules involving generated drift

## Case 1 — Dirty tree contains only known generated drift
Action:
- revert the drift
- re-run `git status`
- if clean, continue package closeout

## Case 2 — Dirty tree contains package files plus generated drift
Action:
- revert the generated drift
- keep package files
- continue normal closeout

## Case 3 — Dirty tree contains ambiguous generated-like changes
Action:
- inspect scope and recent package work
- determine whether the file belongs to the package
- if ambiguity remains real, escalate instead of guessing destructively

## Case 4 — Generated drift reappears after closeout steps
Action:
- reclassify it
- revert again if still non-intentional
- ensure the tree is clean before continuing

The shell must not continue into the next unit with recurring known drift still present.

---

# Rules during resume

Generated drift must also be handled during resume, not only during normal package closeout.

If the shell resumes and sees a dirty tree, it must determine whether the dirty state is:

- unfinished active package work
- completed but uncommitted package work
- known generated drift
- ambiguous mixed state

If drift is the only remaining issue, the shell should clean it automatically and continue.

The user should not need to manually revert drift before `continue` can work.

---

# Rules during hardening passes

Generated drift handling applies equally during hardening passes.

If a hardening pass finishes and the only remaining dirty file is a known drift file, the shell must revert it before pass closeout.

This is required because hardening passes are also supposed to leave behind committed, clean states.

---

# Known-drift list maintenance

The shell may maintain an explicit known-drift list.

Possible forms:
- documented list in shell docs
- internal runtime rule set
- helper artifact listing known recurring generated files

## Requirements for adding a new known-drift file
A file should be added only if:
- it has been observed repeatedly
- it is truly generated/tooling-driven
- it commonly does not belong to package intent
- treating it as drift reduces real operational friction

The shell must not over-broaden this list casually.

---

# Escalation rules for generated drift

The shell should **not** escalate just because known generated drift exists.

Escalation is only appropriate if:
- the file’s role is genuinely ambiguous
- reverting it may destroy real intended package work
- the repo state mixes drift with unresolved unrelated changes in a way the shell cannot safely classify

In those cases, the shell may escalate using the structured escalation template.

But the default is:
- classify
- revert if drift
- continue

---

# Minimum shell commands relevant to drift handling

At minimum, the shell should use:

- `git status`
- `git diff --stat`

Optionally, it may use more targeted inspection commands if needed.

The shell does not need a huge drift-specific toolchain.
It needs disciplined classification and automatic cleanup.

The shell may operationalize generated-drift handling with a short checklist like:

```text

[ ] git status inspected
[ ] git diff --stat inspected
[ ] known drift files identified
[ ] intentional vs non-intentional classification performed
[ ] non-intentional drift reverted

This exact formatting is optional, but the logic is required.

# Final conclusion

The reusable shell must treat recurring generated drift as a first-class operational concern.

Files like `next-env.d.ts` should not repeatedly interrupt package closeout, dirty-tree checks, resume behavior, or hardening completion.

The shell must:
- detect drift
- classify it
- revert it when it is not intentional
- and continue cleanly

That is the generated-drift contract defined here.
